# Vector DB

## Overview
A vector database stores and indexes high-dimensional vectors (embeddings) generated by machine learning models to support fast semantic similarity searches.

## Core Mechanics & How It Works
Queries project text embeddings into multi-dimensional space, performing Approximate Nearest Neighbor (ANN) searches using algorithms like HNSW (Hierarchical Navigable Small World) or IVF-PQ.

## Architecture & Common Patterns
Here are the primary ways this topic is implemented in production systems:
- **RAG Ingestion**:  Extracting, embedding, and indexing document chunks in a vector database.\n- **Hybrid Search**:  Combining vector search with traditional keyword search (BM25) for more accurate results.\n
## Trade-offs & Limitations
- **Operational Complexity**: Implementing this concept increases overall design complexity and setup.
- **Resource Constraints**: It often trades CPU/memory for network/disk resources or vice versa.
- **Tradeoffs**:
  - Calculating distance metrics (cosine, dot product) across millions of dimensions is CPU and memory intensive.\n  - Indices are hard to update dynamically, requiring periodic reconstruction.\n
## System Design Interview Strategy
- **What to Focus On**: Highlight the role of vector databases (such as Pinecone, Milvus, Qdrant) in Retrieval-Augmented Generation (RAG) pipelines, and explain the trade-offs between search accuracy (recall) and latency.
- **Common Scenarios**: Address how this maps to high availability, consistency constraints, or low-latency targets.
